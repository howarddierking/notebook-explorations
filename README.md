# Dealer Analytics Notebooks

A collection of [Marimo](https://marimo.io) reactive notebooks for analytics and performance management across the Dealer technology organization.

## Prerequisites and Setup

Requires [uv](https://docs.astral.sh/uv/) for environment and dependency management.

```bash
uv sync                          # create .venv and install dependencies
uv run marimo edit <notebook>.py # open in interactive editor
uv run marimo run <notebook>.py  # open in read-only presentation mode
```

---

## Architecture and Design Principles

### Reactive Notebook Model

All notebooks are built with Marimo, which executes cells as a reactive dependency graph rather than top-to-bottom. Each cell declares its inputs as function parameters and its outputs via its return value. A UI widget (dropdown, selector) triggers re-execution of every cell that depends on its `.value` — without any callbacks or event handlers. This means the entire display updates automatically when a user changes a filter.

One rule Marimo enforces: a widget's `.value` cannot be read in the same cell that created it. Widgets are always defined in one cell and consumed in a downstream cell.

The final cell of each notebook assembles all UI elements into a single `mo.vstack([...])` expression. This expression is the cell's output — not a return value. Marimo displays the last unassigned expression in a cell; returning a value only exports it to other cells without rendering it.

### Data Layer: CSV as SQL Proxy

Each notebook reads data from CSV files in `data/`. These CSVs are generated by seed scripts (`seed_data.py`, `seed_performance_data.py`) and are intentionally structured to mirror the schemas of SQL tables and views that would back a production deployment. Each seed script contains the corresponding DDL as a docstring so the query shapes are self-documenting.

To connect a notebook to a real data source, replace the `pd.read_csv()` calls with SQL queries against the equivalent tables — the column names and data types remain the same.

**Data files:**

| File | Description |
|---|---|
| `data/mbr_metrics.csv` | Long-format org metrics by month, org level, and metric name |
| `data/people.csv` | Employee roster with org hierarchy and role classification |
| `data/individual_reviews.csv` | Per-person per-period performance reviews with role-specific metric columns |
| `data/individual_goals.csv` | Current-period goals with status and completion percentage |

To regenerate the seed data:

```bash
uv run python seed_data.py               # regenerates data/mbr_metrics.csv
uv run python seed_performance_data.py   # regenerates data/people.csv,
                                         #   individual_reviews.csv,
                                         #   individual_goals.csv
```

### Organization Structure

All notebooks share a common org hierarchy:

```
Financial Services (Division)
└── Dealer (Line of Business)
    ├── DTD        → Platform, Integrations, Analytics
    ├── FMD        → Originations, Servicing, Risk
    ├── CLO        → Lending, Operations, Compliance
    ├── Dealer Core → Identity, API, Data
    ├── DR         → Recovery, Collections, Reporting
    └── Fraud      → Detection, Investigation, Prevention
```

Employees are classified as either individual contributors (`is_leader = 0`) or people leaders (`is_leader = 1`, covering team leads, sub-org leads, and the org lead).

### Visualization

Charts are built with [Altair](https://altair-viz.github.io/) (Vega-Lite). All charts are wrapped in `mo.ui.altair_chart()` for interactivity. Configuration calls (`configure_view`, `configure_axis`, etc.) are always applied to the outermost composed chart, never to individual layers or panels.

## Monthly Business Review (`mbr.py`)

The MBR notebook provides a comprehensive view of the Dealer organization's operational health for the current month, with trend context across the prior two months (January–March 2026).

### How to Use

A single **"View by" dropdown** at the top of the page controls the entire notebook. Selecting **Dealer** shows aggregate data for the whole organization. Selecting any sub-org (DTD, FMD, etc.) switches all charts to show that sub-org's teams. The scorecard always reflects the selected org level.

### Summary Scorecard

A color-coded HTML table showing every metric grouped by category. Each row includes:

- **Red / Yellow / Green threshold ranges** — the criteria used to determine status
- **Current value** — displayed with a colored RAG badge (red/yellow/green)
- **vs. Prior Month** — directionally-aware percent change; favorable moves are green, unfavorable are red

### Deep Dives

Tabbed sections for **Customer**, **Economics**, **Talent**, and **Quality**, each containing a trend chart and a breakdown bar chart for every metric in that category.

- **Trend chart** — monthly values for the selected org across the three-month window; incidents and work type allocation use multi-line and stacked-area formats respectively
- **Bar chart** — current-month values for each child org (sub-orgs when viewing Dealer; teams when viewing a sub-org), color-coded by RAG status

### Metrics

| Category | Metric | Direction |
|---|---|---|
| **Customer** | Deployment Frequency | higher is better |
| | Lead Time | lower is better |
| | Cycle Time | lower is better |
| **Economics** | PRs per Engineer | higher is better |
| | % Time on CTD & RTE | lower is better |
| | Direct Expense Variance | within ±5% is green |
| | AWS Expense Variance | within ±5% is green |

The **Work Type Distribution** deep dive shows the breakdown of engineering time across Investment, Tech Modernization, CTD, and RTE as a stacked chart. The scorecard summarizes this as **% Time on CTD & RTE** (the two demand-driven categories).

## Individual Performance Tracker (`performance.py`)

The performance notebook is designed to support rating calibration conversations. It shows where an individual sits relative to their peers at multiple organizational altitudes — not to track aggregate distributions, but to give the reviewer context for a single person's numbers.

### How to Use

Use the **Organization** dropdown to narrow the **Person** dropdown to a specific org level. Once a person is selected, their profile and metric charts appear. Changing the org selector resets the person selector.

### Profile Card

Displays the selected employee's name, title, current rating (color-coded by band), and a summary grid: level, employee type, team, manager, sub-org, tenure, employee ID, and start date.

### Metric Charts

Each metric renders as a set of three **violin + box plots** side by side — one for each comparison altitude:

- **Team** — all peers of the same role type on the employee's immediate team
- **Line of Business** — all same-role employees within Dealer
- **Division** — all same-role employees within Financial Services

The violin shape shows the density of values across the peer group. The box plot overlaid on each violin shows the quartile distribution (IQR box, median line, whiskers). A single **dashed horizontal line** spans all three panels at the employee's own metric value, making it immediately visible where they fall relative to each peer group.

Role determines which metric set is shown:

**Individual Contributors**

| Metric | Competency | Direction |
|---|---|---|
| Throughput Consistency | Results Focus | higher is better |
| Scope of Impact | Job-Specific Skills | higher is better |
| Cycle Time | Problem Solving | lower is better |
| Change Failure Rate | Judgment | lower is better |
| Review Influence Index | Communication | higher is better |

**People Leaders**

| Metric | Competency | Direction |
|---|---|---|
| AI Tool Penetration | Influence | higher is better |
| SLO Health | Customer Focus | higher is better |
| Knowledge Contribution Rate | Teamwork | higher is better |
| Say/Do Ratio | Results Focus | higher is better |
| Innovation vs Maintenance | Judgment | lower is better |
